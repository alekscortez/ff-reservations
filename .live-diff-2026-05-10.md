# Live side-by-side diff: reservations-new
**Date:** 2026-05-10
**Method:** chrome-devtools-mcp running React (:4200) and Angular (:4300) against the same prod backend, both authenticated with the same staff/admin user, on the same Noche de Reinas event (2026-05-09).

This catches what the static audit missed: the **structural layout** of the page is materially different. The 22-item / 74-item audits both started from "the React port has the same shape, what features are missing?" — that's wrong. The React port has the wrong **shape**.

## Structural deltas

### 1. Event picker is not on this page in Angular
- **Angular:** header reads `New Reservation` + secondary line `Event: Noche de Reinas (2026-05-09)`. The current event is selected from `/events/context/current` (the "current/next" pick). To change events you click "Change Event" at the bottom of the page, which opens a modal. There is no event card grid, no event dropdown, no "past events" shortcut on this page.
- **React:** dominant 2/3/4-column event card grid takes the upper half of the page, with a redundant `<select>` dropdown below, plus a "Past events (5)" button. Forces every staff into picking even though 99% of bookings are for the current/next event.

### 2. Layout is map + list always-split, not toggle
- **Angular:** at all widths I tested (~1100px), map is left and a scrollable table list is right. List has its own header: search input, filters icon, "TABLE LIST" label, and a "111 results" count. No view-toggle pill.
- **React:** below `lg`, the page is single-column with a "Map / List" tab toggle. Above `lg` (which I added in 5.2c/6) the split layout exists but doesn't extend below. At the typical staff laptop width Angular is split, React is single.

### 3. List item shape
- **Angular list rows:** `A01` mono-font, `$40` price right-aligned, status pill ("AVAILABLE" / "PENDING PAYMENT" / "RESERVED"). Disabled rows for non-AVAILABLE.
- **React list rows:** `A01 / $40.00` with a section-colored left border, no status pill, no disabled-state list rows. Status only shows when not AVAILABLE.

### 4. Hold pattern: select-then-confirm vs click-to-hold
- **Angular:** click table (in list or map) → row + map cell highlight, bottom CTA enables. `Hold & Reserve` button at bottom-right confirms. Two clicks, deliberate.
- **React:** click table → immediately hits `POST /holds`. One click, fast but no chance to reconsider.

### 5. Bottom CTA bar
- **Angular:** persistent bottom bar with "Selected: Table A01 $40", "Change Event", "Hold & Reserve" (red, primary). Always visible.
- **React:** no bottom CTA bar. CTAs are inline in the form modal only.

### 6. Booking modal title + form layout
| | Angular | React |
|---|---|---|
| Title | "Reservation Details" | "Customer + payment" |
| First field | Phone (with country combobox left, number right, in one row) | Customer name (full-width row) |
| Second field | Customer Name | Phone (1fr / 120px country split) |
| Total label | "Total Amount Due" | "Total amount due (USD)" |
| Method picker | Square / Cash App / Cash buttons (same as React) | ✓ |
| CTA label | "Confirm & Generate Square Link" (dynamic — changes by method) | "Confirm reservation" (static) |
| Deadline label | "Deadline Time (America/Chicago)" inline | "Deadline time" without tz inline |
| Notice | "Digital payment flow" header + 2-line description | One-line "Reservation creates as PENDING…" |

### 7. Information surfaced

- Angular shows a `4:55` countdown badge top-right of the modal. Live shrinking.
- Angular shows "Default for pending/partial is next day at 12:00 AM (America/Chicago)" as the deadline hint.
- Angular shows a "Digital payment flow" info panel summarizing the flow.

### 8. App chrome differences
- Angular header: brand mark + "Reservation System" wordmark + "Next: Noche de Reinas / MAY 9" pill + "Quick actions" button. Sidebar has 6+6 items grouped under STAFF and ADMIN.
- React header: "Famoso Fuego — Reservations" title + "EN / ES" toggle + email + "Sign out". Sidebar lists items differently (Active Holds / Birthday Packages / Customer CRM / Staff & Admin Users).
- These divergences exist outside reservations-new and won't be fixed there.

## What this means for mitigation

The three big restructures are:

1. **R1 — Move event picker off this page.** Use `/events/context/current` to default the eventDate; render only the current event in the header. Add a "Change Event" modal triggered from the bottom CTA. Past events stay accessible via the modal.
2. **R2 — Make split layout the default.** At all widths ≥ ~768px, render map left and list right. List has its own search + filter trigger + result count, scrolls internally. The view-toggle pill only appears below 768px.
3. **R3 — Two-step hold pattern.** Clicking a table sets `selectedTableId`. The bottom CTA bar becomes the canonical action: "Selected: Table A04 · $40" + "Change Event" + "Hold & Reserve". Only on `Hold & Reserve` does `POST /holds` fire. List items + SVG cells highlight on selection.

After those three, smaller deltas remain:

4. Booking modal title → "Reservation Details"; reorder fields phone-first; modal-internal countdown badge; dynamic CTA label.
5. List rows: render status pill explicitly; disable non-AVAILABLE.
6. Notice copy parity (Digital payment flow header, deadline default text).

## Scope estimate

R1 + R2 + R3 are tightly coupled — they touch the same render tree. Best done as one larger commit (~2 hours) rather than three half-baked partials. After that, the smaller deltas (4–6) are an afternoon's polish.
