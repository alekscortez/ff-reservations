# True line-by-line audit (read end-to-end personally, not delegated)
**Date:** 2026-05-10
**Method:** Read every line of Angular (`reservations-new.ts` 1976 LOC + `.html` 970 + `.scss` 99) and React (`reservation-new.tsx` 2028 + `table-map.tsx` 340) personally. Walked methods 1-by-1 against React equivalents.

The earlier 22-item / 74-item audits caught the surface. This read finds the deep plumbing.

## A. Plumbing the React port doesn't have at all

### A1. `?date` vs `?eventDate` query param
Angular reads `params.get('date')` (reservations-new.ts:191-200). External deep-links use `?date=YYYY-MM-DD`. React reads `?eventDate`. **Any saved bookmarks or shared URLs from the Angular era won't preselect anything in the React port.**

### A2. `holdCreatedByMe` flag — entire concept missing
Angular tracks who created the hold (lines 110, 458, 468, 492, 632, 773, 1847). When it's *not* us — i.e. an existing lock on the table — closing the modal doesn't prompt to release. React always shows the release-confirm regardless of who held the lock. Real bug if staff ever lands on a row whose hold belongs to another session.

### A3. `clearEventSelection()` for "Change Event" CTA
Angular method (lines 378-396) wipes everything: eventDate=null, event=null, tables=[], selectedTable=null, holdId=null, holdExpiresAt=null, polling stops, all panel heights reset. React's "Change Event" button just opens the past-events modal — eventDate is never null after the first auto-pick. The user can't deliberately enter a "no event" state.

### A4. `formErrorMessage()` per-field validation
Angular (lines 1084-1101) returns specific error per invalid field: "Customer name is required", "Phone is required", "Total amount due must be 0 or greater", etc. React just disables the submit button on `!formState.isValid` — no per-field message. Staff hits a disabled button with no explanation.

### A5. `tableCounts()` summary
Angular (lines 1596-1620) computes `{ total, available, hold, pendingPayment, reserved, disabled }`. Used for status badges. React has section stats (A: 65/69) but no breakdown by status.

### A6. `onWindowResize` + `onWindowScroll` HostListeners
Angular (lines 1731-1741) recomputes desktop split height + compact panel height on every resize and scroll. React has no resize listener — split heights are baked in via `max-h-[68vh]` Tailwind classes.

### A7. `scheduleDesktopSplitLayout` + height recalculation
Angular (lines 1743-1811) dynamically sizes:
- `desktopSplitHeightPx` — split panel fits viewport minus sticky header
- `compactPanelHeightPx` — mobile map/list panel fits viewport minus sticky CTA
- `compactBottomInsetPx` — measured from the sticky CTA bar's actual height
- `compactSectionBottomPaddingPx` — pads below the panel so content isn't hidden

Uses `visualViewport.height + offsetTop - rect.top - bottomGap` math + `requestAnimationFrame` debounce + `getBoundingClientRect()` of refs. React: hardcoded `max-h-[68vh]` and `max-h-[70vh]`.

### A8. Status filter granularity (6 values, not a boolean toggle)
Angular `filterStatus` is `'ALL' | 'AVAILABLE' | 'HOLD' | 'PENDING_PAYMENT' | 'RESERVED' | 'DISABLED'` (lines 156-160). React replaces this with two booleans `showAvailable / showUnavailable`. **Lose the ability to filter "show only HOLDs" or "show only PENDING_PAYMENT".** Staff doing operational checks can't isolate a single status.

### A9. Section filter as single-select
Angular `filterSection: string` = `'ALL'` or one section letter (line 161). React uses `sections: string[]` multi-select. Different UX — Angular's pattern matches the rest of the staff app.

### A10. Past events: two separate filters (date + name)
Angular has `pastFilterDate` + `pastFilterName` as separate FormControls (lines 165-166), with a date input and a name input side-by-side, plus a Clear button. React: one combined search input. **Less precise filtering.**

### A11. `exactMatchPhone` + auto-fill at 10 digits
Angular (lines 254-267, 815-823, 965-969):
- Tracks `exactMatchPhone` when search returns a row whose normalized phone matches the typed digits AND digits.length >= 10
- On exact match: auto-fills name + phone, clears matches list, sets `isExactMatch(client)` highlight on the matched row

React: shows match list at 3+ digits, no auto-fill, no exact-match flag, no row highlight.

### A12. CRM "no match" gating to 10+ digits
Angular shows "No match found" only after entering 10 digits (line 259). React shows it at 3+ digits — discourages early typing because every short typed phone shows "no match".

### A13. CRM debounce 250ms vs 350ms
Angular: `debounceTime(250)` (line 235). React: `setTimeout(setDebouncedPhone, 350)` (line 537). Trivial, but matches.

### A14. `clientCreditsTotalRemaining()` + `clientCreditLabel()` with expiry
Angular displays "Total $135.00" header for the credits panel (line 670, method at 834-838). Each credit row shows `$X.XX · Expires May 12, 2026` or `$X.XX · No expiry` (method at 840-844).

React: panel shows just amount, no total, no formatted expiry. Just `{moneyFormatter.format(remaining)}{c.expiresAt ? ` · exp ${c.expiresAt}` : ''}` — raw ISO date.

### A15. `remainingMethod` selector + cash-remainder addPayment
Angular's credit flow (lines 654-707, 906-916, 1140-1170):
- When credit covers partial amount, staff picks `remainingMethod: 'cash' | 'square' | 'client'`
- If 'cash': after credit applies, fires a SECOND `addPayment({ amount: remaining, method: 'cash' })`
- If 'square'/'client': skips second payment, calls `generatePaymentLinkForCurrentFlow()` for the remainder
- `creditNeedsDeadline()` returns true only when remainder > 0 AND remainingMethod !== 'cash'

React: no `remainingMethod` field. Credit applies, then the post-create panel handles the remainder via Square link (manual). Staff can't pre-declare cash remainder; doesn't auto-fire the second payment.

### A16. `autoSquareLinkSms.sent` response check
Angular checks `created?.autoSquareLinkSms?.sent` (lines 711, 731) — if backend already auto-sent SMS, skips the link-generation panel and goes straight to "Done". React ignores this field entirely. Result: even when backend already SMS'd the link, React's modal asks staff to manually generate again.

### A17. `finishReservationFlow()` — full reset
Angular (lines 1506-1527):
- clearActiveHoldSession()
- resetCreatedReservationState()
- selectedTable=null, selectedTableId=null
- allowCustomDeposit=false, paymentDeadlineEnabled=false
- showReservationModal=false
- form.reset({ ...defaults })
- phoneCountry='US'
- clearClientCreditsState()

React's `onAnother` (in PostCreatePanel) calls a similar reset but doesn't reset `phoneCountry` to 'US' (back to inferred default), doesn't clear credit state explicitly — since the underlying credits query refires when phone changes, but there's no `creditsLookupKey` invalidation.

### A18. `selectTable()` clearActiveHoldSession() + amount reset
Angular's `selectTable(t)` (lines 448-464) does:
- clearActiveHoldSession() — discards any persisted session for a different table
- selectedTable = t, selectedTableId = t.id
- resetCreatedReservationState()
- holdId / holdExpiresAt / holdCountdown reset
- holdCreatedByMe = false, showReleaseConfirm = false
- allowCustomDeposit = false
- amountDue = price (this is where the table price prefills, NOT after createHold)
- applyPaymentDefaultsForCurrentMethod()

React's `handleTableSelect(table)` (lines 398-401): just `setSelectedTableId((cur) => cur === table.id ? null : table.id)`. Doesn't reset amount, doesn't reset payment defaults, doesn't clear hold state. **The amount-due prefill happens via a different effect after `heldTable` is set, which is post-hold-creation.** Different timing.

### A19. `createHold()` sends customerName + phone + phoneCountry
Angular's createHold body (lines 480-487):
```
{ eventDate, tableId, customerName, phone, phoneCountry }
```
React's body: `{ eventDate, tableId }` only. The hold lock row in DDB will lack contact info. If the hold gets converted to a reservation later, those values come from the reservation create payload — but Angular's pattern means staff *could* see who's holding what mid-flow.

### A20. Phone E.164 normalization on submit
Angular calls `normalizePhoneToE164(phone, country)` before BOTH createHold and confirmReservation (lines 474-477, 559-562). React submits the form's raw `phone` field. The backend likely normalizes anyway, but Angular surfaces `'Phone must be a valid US or MX number'` client-side if normalization returns empty.

### A21. `nextDate(eventDate)` — deadline default uses event date
Angular's `setDefaultPaymentDeadline()` (lines 995-1001) calls `nextDate(eventDate)` to compute the deadline date — i.e. the day AFTER the event. React uses `nextDayDateString()` which is browser-today + 1. **For an event 4 days from now, Angular sets deadline to event-day+1; React sets it to tomorrow.** Different default.

### A22. `setDefaultPaymentDeadline` time uses configured defaults
Angular sets `paymentDeadlineTime` to `formatHm(defaultPaymentDeadlineHour, defaultPaymentDeadlineMinute)` from /events/context/current. React sets time once on hold creation if not already touched, but defaults to literal `'00:00'` in the form.

### A23. Hold timer `m:ss` format
Angular: `${min}:${String(sec).padStart(2,'0')}` → "4:55". React: `${Math.floor(remainingSec / 60)}m ${String(remainingSec % 60).padStart(2, '0')}s` → "4m 55s". Trivial, but visible.

### A24. Status pill labels on list rows
Angular renders the status label on EVERY list row (template lines 333-340 for compact, 541-548 for desktop):
- AVAILABLE → "Available"
- PENDING_PAYMENT → "Pending Payment"
- HOLD → "HOLD" / RESERVED → "RESERVED" / DISABLED → "DISABLED"

Plus border/bg colors per status (lines 300-305, 508-513): accent for HOLD, warning for PENDING_PAYMENT, danger for RESERVED, brand-50 for DISABLED.

React shows the status pill ONLY when not AVAILABLE, and uses a plain muted bg for all non-AVAILABLE. Loses the at-a-glance status differentiation.

### A25. Mobile "Tap a green table to start reservation" hint
Angular shows this hint above the map on mobile when MAP mode is active (template line 132). React doesn't.

### A26. Modal `pb-28` on mobile keeps submit visible
Angular's modal class (line 587) is `p-4 pb-28 md:p-6` — extra bottom padding so the form's last field isn't hidden under the sticky mobile CTA bar. React modal has `p-5` everywhere; on mobile the bottom of the form sits behind the sticky bar.

### A27. Modal max-height `92dvh`
Angular: `md:max-h-[92dvh]` (line 587). React modal: no max-height. Tall forms can exceed the viewport; React relies on `overflow-y-auto` on the outer fixed div which works but loses the "card" feel.

### A28. `linkCollectionTitle()` for Cash App
Angular: `currentLinkMode() === 'client' ? 'Cash App Link' : 'Square Link'` (line 1294). React: hardcoded "Square payment link" in PostCreatePanel. **For a Cash App method reservation, React still says "Square payment link".**

### A29. `generatePaymentLinkForCurrentFlow` handles Cash App
Angular (lines 1186-1212) calls `reservationsApi.createCashAppLink(...)` for the `'client'` method — distinct endpoint from Square. React only has `useCreateSquarePaymentLink` in PostCreatePanel. **Cash App link generation isn't wired in the new-reservation flow.** The reservation will create successfully, but the link panel can't generate a Cash App link.

### A30. Workspace scroll lock when eventDate is set
Angular's `syncWorkspaceScrollLock()` (lines 1637-1651) locks `<body>` AND `<html>` overflow when `eventDate` is set AND no modal is open. The intent: the table picker takes over the viewport, no document scrolling.

React only locks during modal open. With eventDate set + no modal, the React page scrolls normally. Different mental model — React treats the picker as a regular page, Angular treats it as a fullscreen workspace.

### A31. `paymentMethodOptions` uses `'client'` value for Cash App
Angular's enum: `'cash' | 'square' | 'client'` (lines 131-135). React uses `'cash' | 'square' | 'cashapp'`. Backend accepts `'cashapp'` (per `services-reservations-holds.mjs:1877`), so Angular's `'client'` → `'cashapp'` mapping happens via `toCreatePaymentMethod` (line 1146-1152). Cosmetic difference but the React form value is closer to backend.

### A32. `(mousedown)` AND `(click)` on payment method buttons
Angular wires both events (template lines 731-732) for touch-input reliability on mobile Safari. React: just `onClick`. Should not matter on modern Safari but Angular's belt-and-suspenders.

### A33. Filter button shows count badge
Angular renders a numeric badge on the filter button when `activeFiltersCount() > 0` (template lines 168-173). React shows a binary dot (active or not). Loses the count.

### A34. `statusFilterLabel()` / `sectionFilterLabel()` / `filtersButtonLabel()`
Angular methods (lines 1372-1396) format the filter button as e.g. `Filters (2)` with active filter labels. React: just "Filters" + a dot.

### A35. `applyPaymentDefaultsForCurrentMethod()` cleanly groups method-aware logic
Angular has a single private method (lines 1444-1468) that resets state when payment method changes. React has the same logic spread across 3 useEffects (lines 359-380), making it harder to follow when one effect overwrites another's value.
